# SudoMode — AI Context File

## Project Identity

**SudoMode** is a governance & authorization middleware proxy for AI agents. It acts as the "missing `sudo` command" — intercepting high-risk autonomous agent actions, evaluating them against configurable YAML policies, and enforcing human-in-the-loop approval before execution. Agents send **intents** (what they want to do), and SudoMode holds the keys.

**GitHub:** [github.com/numcys/sudomode](https://github.com/numcys/sudomode)  
**License:** MIT

---

## Architecture Overview

```
AI Agent (SDK Client)
    │  Intent Request (resource, action, args)
    ▼
SudoMode Proxy Server (FastAPI)
    ├── Policy Engine ──► YAML rules (first-match-wins, default DENY)
    ├── Request Store  ──► In-memory dict (pending/approved/rejected)
    ├── Slack Service   ──► Optional webhook notifications
    └── REST API (/v1/govern, /v1/requests/*)
              │
    ┌─────────┴──────────┐
    ▼                    ▼
React Dashboard     SDK Long-Polling
(approve/reject)    (auto-waits for decision)
```

**Three possible decisions:** `ALLOW` → execute immediately · `DENY` → block · `REQUIRE_APPROVAL` → pause, notify humans, wait for decision.

| Method | Route                           | Purpose                                  |
| ------ | ------------------------------- | ---------------------------------------- |

| POST   | `/v1/requests/{id}/reject`      | Reject a pending request                 |

## Core Components

### Policy Engine (`server/app/core/engine.py`)
- **Class:** `PolicyEngine`
- Loads rules from `policies.yaml` (YAML)
- Evaluates in order: **first-match-wins**
- Supports exact match and wildcard (`*`) for resource/action
- Conditional expressions via `simpleeval` (e.g., `args.amount > 50`)
- **Default: DENY** when no rules match (fail-safe)

### Request Store (`server/app/core/store.py`)
- In-memory dict-based storage
- Functions: `add_request()`, `get_request()`, `get_all_requests()`, `update_request_status()`, `remove_request()`
- Statuses: `PENDING` → `APPROVED` or `REJECTED`

### SDK Client (`sdk/sudomode/client.py`)
- **Class:** `SudoClient`
- `check(resource, action, args)` → low-level governance call
- `execute(resource, action, args)` → full workflow with long-polling for approvals
- Raises `PermissionError` for DENY/REJECTED
- Returns `True` for ALLOW/APPROVED
- Context manager support (`with SudoClient() as sudo:`)

### Dashboard (`dashboard/src/App.jsx`)
- Polls `/v1/requests` every 2 seconds
- Displays pending requests as cards with approve/reject buttons
- Dark theme, cyberpunk/security aesthetic
- Shows recent activity (approved/rejected history)

## Data Models

```python
# GovernanceRequest (Pydantic)
resource: str          # e.g., "database", "stripe.charge"
action: str            # e.g., "read", "delete", "charge"
args: Dict[str, Any]   # e.g., {"amount": 100}

# GovernanceResponse (Pydantic)
status: Literal["ALLOW", "DENY", "REQUIRE_APPROVAL"]
reason: str
request_id: Optional[str]  # Present when REQUIRE_APPROVAL
```

## Policy Format (`policies.yaml`)

```yaml
rules:
  - name: "rule_name"
    description: "Human description"
    resource: "stripe.charge"       # or "*" for wildcard
    action: "charge"                # or "*" for wildcard
    condition: "args.amount > 50"   # optional, uses simpleeval
    decision: "REQUIRE_APPROVAL"    # ALLOW | DENY | REQUIRE_APPROVAL
    reason: "Explanation for decision"
```
Rules are evaluated top-to-bottom; first match wins. Always end with a catch-all default deny rule.

## Agent Instructions

When working on this codebase:

1. **Identify scope.** Determine which component is being modified — Backend Server (`server/`), SDK (`sdk/`), Dashboard (`dashboard/`), or Config/Policies.
2. **Do not over-engineer.** This project values simplicity and clarity. Keep abstractions minimal.
3. **Preserve the fail-safe pattern.** Default to DENY. Never silently allow unmatched requests.
4. **Write clean, commented code.** Use type hints (Python), docstrings, PEP 8 style. Use `const` in JS, functional React components.
5. **Use print/log statements** for visibility — the terminal output should clearly show `ACCESS DENIED`, `APPROVED`, `WAITING FOR APPROVAL`, etc.
6. **Do not edit files outside the scope of the immediate task.**
7. **Never create infinite loops without a breakout condition** — protect compute resources.
8. **CORS origins** are explicitly whitelisted in `server/app/main.py` — update if frontend port changes.
9. **Slack integration is optional** — controlled via `SLACK_WEBHOOK_URL` in `server/.env`. Notifications must fail gracefully (never block a governance request).
10. **Storage is in-memory only** — no persistence across server restarts (by design for MVP).
